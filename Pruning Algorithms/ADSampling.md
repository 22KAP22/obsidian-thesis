ADSampling (https://arxiv.org/pdf/2303.09855) is a method to prune vectors during search. The goal is to reduce the amount of time used to compute distances.

High-level explanation of how it works:
Most indexes with their search algorithms spend most of their time doing Distance Computation Operations (DCOs). Most of these operations are done on objects/vectors, that will be discarded anyway, since they're too far away from the query. The goal of ADSampling is to reduce the time spent on DCOs by doing a projection with an orthogonal Matrix and then sampling only a few of the dimensions. With high confidence it can early deduce that a given vector isn't close to the query vector and can thus prune it/not care about it anymore. Since most of the time is spent on distance computations and most of the distance computations are done on vectors that in the end aren't even candidates, this greatly improves the efficiency of search algorithms.

Interestingly this method doesn't have any assumptions on the data it works on and thus can be in cooperated into any search algorithm. In order for this to be effective though, the search algorithm needs to have a lot of vectors that it needs to investigate/do DCOs on.

A little bit more technicality:
There will never be false positives, since it only accepts a vector if and only if all the dimensions are considered for the distance computation. The true distance is evaluated at this point. There can be false negatives though. If it happens that a vector that is in reality a candidate vector it can be pruned early. The probability of this happening can be steered through a parameter. This way one can choose between higher-accuracy or faster search tuning this parameter. This parameter essentially controls the confidence it needs to discard a vector. They call this parameter ε0(Epsilon0).

Using these random rotations with an orthonormal(or orthogonal? not sure which one it is) projection, there is a theorem/lemma that bounds the projected distance to the real one. This contains the ε parameter. They then use this formula to show that the probability of having a false negative is very low, respectively depends on the ε parameter, which in practice in such a way that the probability of it happening is very low.
During index build, before answering any query the entire dataset/all data points need to be projected with the random projection matrix. This needs to be done only once for the entire set. New points then simply need to be projected with the same projection. Note that this should lead to the same indexes being built or rather the indexes for the original data points should correspond to the ones with the projected data points. They argue that the cost of this transformation can be amortized, since it can be reused for all future queries. Similarly the incoming queries also need to be projected. They argue again that this cost can be amortized, since it is needed for all DCOs during the search, which, as they show with empirical data, dominate the running time of most state-of-the-art search algorithms.
Distance computation then works as following:
Incrementally the dimensions will be retrieved from the transformed query and the data point it computes the distance of. For each iteration it computes the 'partial' distance. If it's greater than a threshold it gets classified as a negative (meaning its distance is too great to be a potential close neighbor) and this data point doesn't need to be considered anymore during the rest of the search. The reason why this works is that considering the first d dimensions of the transformed data point and query can be interpreted as also being randomly projected. Thus the same lemmas/theorems hold here too. The threshold needs to be scaled by the amount of dimensions considered, but the rest almost stays the same. This way we can slowly introduce more and more dimensions until we can confidently say it's either not a candidate, or if all dimensions were inspected that it truly is a candidate.

The authors of the paper also included versions of HNSW and IVF. For IVF it's simply the memory layout they changed (PDX does similar things). HNSW it's a little bit more complicated, since they use the sets to keep track of neighbors to check for candidates and actual candidates  in different ways. The idea here is to have tighter thresholds for the ADSampling algorithm further improving the efficiency with very small cost to accuracy.
